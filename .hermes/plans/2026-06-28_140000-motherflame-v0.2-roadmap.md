# Motherflame v0.2 — "From demo-real to production-real"

> **For Hermes:** Use subagent-driven-development to implement phase-by-phase. Each task is TDD + commit. Canonical test = `python3.11 -m pytest tests/` via terminal. Identity = `opelpleple <30042054+opelpleple@users.noreply.github.com>` only.

**Goal:** Take Motherflame from "works in a demo / small brain" to "trustworthy at real-company scale" by closing the four honest limits we surfaced while building v0.1: keyword-only retrieval, shared-key identity, no live connectors, and lossy capture.

**Architecture:** Stay pure-stdlib in core; every heavy capability (embeddings, OAuth, remote sources) plugs in through an existing or new *interface* so the OSS core never hard-depends on a service. TrustFinance-specific plugs live in a fork, never in core.

**Current state (v0.1, shipped):** 19 modules · 115 tests · CI green (3.9/3.11/3.12) · features: conflict resolution (claims→canonical), trust scoring, temporality, review queue, connector + retrieval *interfaces*, documents, web research, create/join/doctor/team/splash, MCP (9 tools), agent team-setup tools.

## DECIDE — priority order (highest leverage first)

| # | Initiative | Why now | Honest limit it closes |
|---|---|---|---|
| **P1** | **Semantic retrieval** (pluggable) | Brain is useless at scale if it can't *find* the right fact. Interface already exists — just needs a real retriever. Highest user-felt impact. | "keyword retrieval doesn't scale past ~hundreds of facts" |
| **P2** | **Per-member identity + roles** | Today anyone with the Flame Key is anyone. Blocks real-team trust + owner authority is spoofable. | "identity is a self-declared string; Flame Key = shared access" |
| **P3** | **Live connectors** (Slack/Notion/Drive) | Coverage is the value prop. Interface exists; only `local_files` is real. | "connector interface has no real remote source" |
| **P4** | **Incremental capture** (watch / hooks) | Brains rot. Capture context as work happens instead of one-off harvests. | "harvest is a manual one-shot; brain goes stale" |
| **P5** | **Sync hardening at scale** | Documents bloat brain.json; concurrent team pushes risk merge pain. | "git blob grows, merge is naive union" |

**Sequencing rationale:** P1 first (it makes every other feature more useful — more data is only good if retrievable). P2 second (unblocks trustworthy multi-person use, which P3/P4 amplify). P3/P4 add data; do them *after* retrieval + identity can handle it. P5 is continuous hardening, slot in as brain grows.

# Phase P1 — Semantic Retrieval (flagship)

**Objective:** A real semantic retriever that plugs into the existing `retrieval.BaseRetriever` interface, ranking facts + document chunks by meaning, not keywords — without forcing an embedding dependency into core.

**Design:** Embeddings come from a pluggable `EmbeddingProvider`. Ship two: (a) `OpenAIEmbedding` (uses the user's existing key, cloud) and (b) `HashingEmbedding` (deterministic local fallback, zero-dep, lower quality). Vectors cache in `brain["embeddings"]` keyed by content hash so we only embed new/changed items. Cosine similarity in pure Python (numpy optional, stdlib `math` fallback).

**Files:**
- Create: `motherflame/embeddings.py` (providers + cache)
- Create: `motherflame/retrievers/semantic.py` (SemanticRetriever)
- Modify: `motherflame/retrieval.py` (register semantic, expose `set_active`)
- Modify: `motherflame/conflicts.py:ensure_layers` (add `embeddings` store)
- Modify: `motherflame/core.py` (`config set retrieval semantic`, reindex command)
- Test: `tests/test_embeddings.py`, `tests/test_semantic_retrieval.py`

# Phase P2 — Per-Member Identity + Roles

**Objective:** Make "who said this" verifiable and give owners real authority, without standing up a server (stay self-hostable).

**Design (DECIDE):** Lightweight asymmetric identity, not full SSO. Each member generates an Ed25519 keypair locally (`motherflame whoami --init`). Their public key + display name go into `brain["members"]`. Claims are *signed* with the private key; `verify_claim`/owner checks validate the signature. The Flame Key still encrypts (confidentiality); signatures add authenticity. Roles (`owner`/`editor`/`viewer`) gate which claims win and who can `/resolve`.

**Files:**
- Create: `motherflame/identity.py` (keygen, sign, verify — uses `cryptography` Ed25519, already a dep)
- Modify: `motherflame/conflicts.py` (store `member_id` + `sig` on claims; trust weights signed > unsigned)
- Modify: `motherflame/core.py` (`whoami --init`, `members` view, role in `/owner`)
- Test: `tests/test_identity.py`

**Tasks (bite-sized):**
- P2.1 Ed25519 keygen + local keystore (`~/.motherflame/identity.json`, 0600). Test: deterministic verify.
- P2.2 Sign claims on write; carry `member_id`+`sig`. Test: tampered value fails verify.
- P2.3 `brain["members"]` registry (pubkey+name+role); `members` command. Test: join adds member.
- P2.4 Trust weighting: signed-by-known-member > unsigned > legacy. Test: resolution prefers signed.
- P2.5 Roles gate `/resolve` + owner authority. Test: viewer can't resolve.

**Honest scope:** This is *authenticity within a team that shares the Flame Key*, not identity-provider SSO. Document clearly: it stops impersonation among members, not a malicious Flame-Key holder. Full SSO/OIDC stays a fork concern.

**Verification:** Two-member sim — member B can't forge a claim as member A; owner-only resolve enforced.

# Phase P3 — Live Connectors

**Objective:** Real remote sources on the existing `connectors.BaseConnector` interface so coverage isn't limited to local files.

**Design:** Each connector is opt-in, uses the user's own token, fetches → yields `Document`s → same harvest/confirm pipeline (nothing auto-enters the brain). Ship in order of leverage:
- P3.1 **Notion** connector (token + database/page IDs) — structured, high-signal.
- P3.2 **Google Drive** connector (Docs/Sheets export to text).
- P3.3 **Slack** connector (channel history → context) — TrustFinance already has Slack creds; the *connector* is core, the TF wiring is fork.

**Files:** `motherflame/connectors/{notion,gdrive,slack}.py`; register in `connectors.py`; `motherflame harvest --source notion` dispatch; per-connector tests with mocked HTTP.

**Tasks per connector (repeat):** auth check → fetch (mocked in tests) → map to `Document` → register → CLI wire → confirm-flow reuse → commit.

**Honest scope:** Remote connectors send fetched content to the LLM on harvest (same privacy caveat as file harvest). Keep them opt-in; never background-sync without consent.

# Phase P4 — Incremental Capture

**Objective:** Keep the brain fresh by capturing context as work happens, not via one-off harvests.

**Design (DECIDE — start narrow):**
- P4.1 **`motherflame watch <folder>`** — poll file fingerprints (reuse `filestate.json`), re-harvest only changed files, stage to review queue (never silent-write). Foreground process, Ctrl-C to stop.
- P4.2 **Git post-commit hook** template — on commit, harvest changed `*.md` docs in the repo into the brain (opt-in install via `motherflame hooks install`).
- P4.3 **Staleness surfacing** — `doctor`/`brain` flag facts older than a configurable TTL ("12 facts haven't been reconfirmed in 90d").

**Files:** `motherflame/watch.py`; hook template under `motherflame/templates/`; `core.py` dispatch; tests with simulated file changes.

**Honest scope:** No daemon/background magic in core — watch is an explicit foreground command; hooks are opt-in. Avoids surprise data egress.

# Phase P5 — Sync Hardening (continuous)

**Objective:** Keep zero-knowledge sync robust as the brain grows (documents make blobs big; teams push concurrently).

**Tasks (slot in as needed):**
- P5.1 **Split documents out of brain.json** into per-doc encrypted blobs (`cloud/docs/<id>`), so the core brain stays small and merges stay cheap. Migration keeps old brains working.
- P5.2 **Three-way merge for claims** — currently union; add detection of "same member re-signed a key with a new value" to avoid duplicate claims piling up.
- P5.3 **Push preflight** — `team`/`doctor` warn "you're N behind, pull first" before a push that would conflict (extend existing `check_remote`).
- P5.4 **Brain size budget** — warn when brain.json crosses a size threshold; suggest doc-split / pruning tombstoned claims.

## Cross-cutting: things to NOT do (YAGNI / scope guards)

- ❌ No web dashboard yet — CLI + MCP is the surface; a dashboard is a separate product, not v0.2.
- ❌ No multi-brain/federation — one org = one brain until single-brain UX is solid.
- ❌ No vector DB dependency in core — embeddings cache in brain.json / local files; pgvector is a *fork* (TrustFinance).
- ❌ No telemetry/analytics — privacy-first; don't add phone-home.
- ❌ Keep every TrustFinance-specific value (Slack wiring, finance vocab, Thai content, Supabase) in the TF fork, never core.

## Risks & open questions

- **Embedding cost/privacy (P1):** OpenAI embeddings send content out + cost money. Mitigation: HashingEmbedding default-safe; require explicit `config set retrieval semantic` opt-in; document clearly. Open Q: ship a local sentence-transformers provider as optional extra, or leave to fork?
- **Identity UX friction (P2):** keypair init adds a step. Mitigation: auto-init on first write with a one-line notice; signatures are best-effort (unsigned still works, just lower trust).
- **Connector maintenance (P3):** third-party APIs drift. Mitigation: thin connectors, mocked tests, opt-in; let the fork own the ones it depends on.
- **Migration safety (P5.1):** moving documents out of brain.json must not lose data on old brains. Mitigation: read-both-write-new migration + a test that loads a v0.1 brain.
